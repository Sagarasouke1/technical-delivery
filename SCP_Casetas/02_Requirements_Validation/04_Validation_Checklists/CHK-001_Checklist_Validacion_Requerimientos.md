# CHK-001 - Checklist de Validación de Requerimientos

## Objetivo
Verificar que la definición funcional del proyecto SCP_Casetas cuente con el nivel mínimo de completitud, claridad, trazabilidad y control antes de pasar a análisis técnico, diseño, desarrollo y pruebas.

---

## 1. Validación General del Proyecto

- [ ] El nombre del proyecto está definido.
- [ ] El objetivo general del proyecto está documentado.
- [ ] La problemática actual está documentada.
- [ ] El valor de negocio está identificado.
- [ ] Las áreas solicitantes e impactadas están definidas.
- [ ] El Product Owner está identificado.
- [ ] La metodología del proyecto está definida.

---

## 2. Validación de Alcance

- [ ] Existe una definición de alcance incluido.
- [ ] Existe una definición de fuera de alcance.
- [ ] El alcance funcional está alineado con la necesidad del negocio.
- [ ] El alcance no invade procesos no contemplados actualmente.
- [ ] Se identificaron dependencias del proyecto.

---

## 3. Validación de Usuarios

- [ ] Los tipos de usuario están identificados.
- [ ] Las funciones de cada tipo de usuario están documentadas.
- [ ] El nivel técnico esperado por tipo de usuario está definido.
- [ ] Existen responsables operativos y administrativos claramente diferenciados.

---

## 4. Validación de Requerimientos Funcionales

- [ ] RF-01 Importación de Archivos está definido.
- [ ] RF-02 Procesamiento de Información está definido.
- [ ] RF-03 Validación por Viaje está definido.
- [ ] RF-04 Identificación de Casetas No Autorizadas está definido.
- [ ] RF-05 Validación de Costos está definido.
- [ ] RF-06 Detección de Cobros Duplicados está definido.
- [ ] RF-07 Gestión de Incidencias está definido.
- [ ] RF-08 Aclaraciones Operativas está definido.
- [ ] RF-09 Reporte de Inconsistencias y Resultados está definido.
- [ ] RF-10 Trazabilidad de Validaciones está definido.

---

## 5. Validación de Requerimientos No Funcionales

- [ ] Integridad de datos documentada.
- [ ] Usabilidad documentada.
- [ ] Disponibilidad documentada.
- [ ] Trazabilidad documentada.
- [ ] Seguridad documentada.
- [ ] Reportería y análisis documentados.

---

## 6. Validación de Reglas de Negocio

- [ ] Existen reglas de negocio documentadas.
- [ ] Las reglas de negocio son coherentes con la operación.
- [ ] Las reglas de negocio cubren validación de rutas, costos, incidencias y duplicados.
- [ ] Las reglas de negocio son entendibles por negocio y por TI.

---

## 7. Validación de Criterios de Aceptación

- [ ] Existen criterios de aceptación para importación.
- [ ] Existen criterios de aceptación para validación.
- [ ] Existen criterios de aceptación para incidencias.
- [ ] Existen criterios de aceptación para reportes.
- [ ] Existen criterios de aceptación para trazabilidad.
- [ ] Los criterios son verificables mediante pruebas.

---

## 8. Validación de Riesgos y Dependencias

- [ ] Los riesgos iniciales están identificados.
- [ ] Las mitigaciones iniciales están documentadas.
- [ ] Las dependencias principales están identificadas.
- [ ] Se reconoce dependencia del formato de archivos del portal PASE.
- [ ] Se reconoce dependencia de catálogos de casetas y tarifas.

---

## 9. Validación de Trazabilidad

- [ ] Existe relación entre HU y RF.
- [ ] Existe relación entre RF y módulos funcionales.
- [ ] Existe relación entre RF y pruebas.
- [ ] Se planea una matriz de trazabilidad formal.
- [ ] La documentación actual permite avanzar a diseño.

---

## 10. Resultado de la Revisión

### Estatus de revisión
- [ ] Completo
- [ ] Completo con observaciones
- [ ] Incompleto
